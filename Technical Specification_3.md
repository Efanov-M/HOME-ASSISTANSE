# Technical Specification_3.md

# ТЗ 3 — Stabilization of Auth Core для проекта Home Secretary

## 1. Смысл этапа

Этот этап не добавляет новый бизнес-модуль.

Его задача — довести уже собранный блок:

- Auth
- Users
- Families
- FastAPI API
- запуск приложения

до действительно рабочего состояния уровня MVP.

То есть сейчас правильный следующий шаг — не переходить к задачам, календарю или документам, а закрыть технологический и эксплуатационный долг текущего auth-ядра.

---

## 2. Почему нужен отдельный этап стабилизации

По текущему состоянию проекта фундамент уже собран:

- есть `core/config.py`
- есть `core/database.py`
- есть `core/security.py`
- есть ORM-модели
- есть repository-слой
- есть `auth_service.py`
- есть `schemas/`
- есть `api/auth.py`
- есть `main.py`

Но при этом auth-ядро ещё не считается полностью завершённым, потому что остаются проблемы не уровня архитектурной идеи, а уровня реальной работы приложения.

Сейчас проект уже вышел из стадии “чернового кода” и перешёл в стадию, где важны:

- стандартный запуск
- корректные HTTP-ответы
- отсутствие runtime-падений на обвязке
- согласованность контрактов между API и service
- ручной прогон всех auth-сценариев

## 7. Архитектурное правило этапа

Всё, что делается на этом этапе, должно оставаться внутри уже принятой цепочки:

API -> Service -> Repository -> DB

Где:

- API принимает HTTP-запрос
- Service решает сценарий
- Repository работает с БД
- DB хранит данные

Запрещено:

- переносить SQLAlchemy-query в API
- переносить JWT-логику в API
- смешивать `main.py` и бизнес-логику
- решать runtime-проблемы хаотическими костылями в разных слоях

# Блок A. Стабилизировать сервисный слой

## Цель блока

Убедиться, что `auth_service.py` работает как единый, согласованный сервисный слой и не падает на обвязке.

## Фокус

`backend/app/services/auth_service.py`

## Нужно сделать

### A.1. Исправить декоратор `@transactional`

Сейчас декоратор берёт `db` через `args[0]`.

Это хрупко, потому что API вызывает сервисы в стиле:

- `login_user(db=db, ...)`
- `register_user(db=db, ...)`
- `logout_user(db=db, ...)`

То есть через keyword arguments.

Нужно сделать так, чтобы декоратор корректно работал:

- и с positional arguments
- и с keyword arguments

### A.2. Проверить сигнатуры всех service-функций

Нужно убедиться, что API вызывает сервисы с теми же именами аргументов и в той же структуре, которые реально объявлены в `auth_service.py`.

Особенно важно проверить:

- `register_user`
- `login_user`
- `refresh_access_token`
- `logout_user`
- `get_current_user`
- `request_password_reset`
- `confirm_password_reset`

### A.3. Нормализовать контракты возврата

Нужно проверить, что каждая сервисная функция возвращает ровно то, что ожидает API-слой.

Примеры:

- `login_user` возвращает пару токенов
- `refresh_access_token` возвращает только access token
- `logout_user` возвращает `bool`
- `request_password_reset` возвращает message
- `confirm_password_reset` возвращает message
- `get_current_user` возвращает ORM-пользователя

### A.4. Проверить согласованность `validate_*` функций

Нужно убедиться, что вспомогательные валидаторы:

- принимают ожидаемые типы
- возвращают ожидаемые типы
- выбрасывают согласованные ошибки
- не ломают вызывающий сценарий

### A.5. Проверить транзакционную модель

Нужно подтвердить, что:

- основная бизнес-логика живёт в одной транзакции
- error-path audit пишется отдельно там, где это уже задумано
- repositories не делают финальный commit самостоятельно
- rollback не разрушает общую логику сценария

## Критерий готовности блока A

Блок считается закрытым, если:

- любой endpoint может вызвать соответствующий service без runtime-падения на транзакционной обвязке
- сигнатуры API и service согласованы
- возвраты service-функций согласованы с response-моделями API
- auth-flow не падает на инфраструктурных мелочах уровня `args[0]`, несовпадения имён аргументов и контрактов

# Блок B. Нормализовать HTTP error handling

## Цель блока

Сделать так, чтобы штатные auth-ошибки возвращались как корректные HTTP-ответы уровня `4xx`, а не как `500 Internal Server Error`.

## Фокус

- `backend/app/api/auth.py`
- `backend/main.py`
- `backend/app/core/errors.py`

## Текущее состояние

Сервисный слой уже выбрасывает:

- `AuthError`
- `ValueError`

Но API-слой сейчас вызывает сервисы напрямую, и глобальной схемы преобразования этих ошибок в HTTP-ответы нет.

Из-за этого типовые ситуации вроде:

- неверный пароль
- несуществующий refresh token
- просроченный reset token
- пользователь заблокирован
- невалидный access token

могут уходить в `500`, хотя это нормальные контролируемые ошибки клиента.

## Нужно сделать

### B.1. Принять единую схему обработки ошибок

Нужно выбрать одно место преобразования доменных ошибок в HTTP-ответы:

- либо в каждом endpoint
- либо через глобальные exception handlers

На этом этапе нужно выбрать один подход и использовать его единообразно.

### B.2. Разложить ошибки по базовым HTTP-статусам

Минимально должны использоваться:

- `400 Bad Request`
- `401 Unauthorized`
- `403 Forbidden`
- `404 Not Found`
- `409 Conflict` — там, где это логически оправдано

### B.3. Не допускать `500` для штатных auth-сценариев

Если пользователь ввёл неверный пароль или прислал просроченный token, это не внутренняя ошибка сервера.

Такие кейсы должны быть обработаны как ожидаемое поведение.

## Критерий готовности блока B

Блок считается закрытым, если:

- `AuthError` и `ValueError` больше не приводят к случайным `500`
- API возвращает контролируемые `4xx` на штатных auth-ошибках
- схема обработки ошибок выбрана одна и используется последовательно

# Блок C. Нормализовать install/run path

## Цель блока

Сделать проект запускаемым по понятной и воспроизводимой инструкции без ручных костылей.

## Фокус

- `backend/requirements.txt`
- `backend/main.py`
- `backend/app/core/config.py`
- `backend/Dockerfile`
- `docker/docker-compose.yml`
- общая схема запуска проекта

## Нужно сделать

### C.1. Проверить зависимости

Нужно убедиться, что `requirements.txt` действительно содержит всё необходимое для запуска auth MVP.

Минимально проверяются зависимости для:

- FastAPI
- Uvicorn
- SQLAlchemy
- psycopg
- PyJWT
- pwdlib
- pydantic
- python-dotenv

### C.2. Проверить стандартный запуск backend

Нужно определить и зафиксировать штатный способ запуска приложения.

Например:

- через `uvicorn`
- из корня backend
- с понятным путём до `app`

Главная цель — исключить ситуацию, когда проект запускается только “если помнить особую магию”.

### C.3. Проверить конфиг

Нужно подтвердить, что env-конфигурация читается:

- из одного места
- по одной понятной схеме
- без дублирования источников настроек

### C.4. Проверить импортную консистентность

Нужно убедиться, что:

- внутренние импорты проекта согласованы
- точка входа не падает на import-time
- проект стартует в штатном режиме

## Критерий готовности блока C

Блок считается закрытым, если:

- backend можно поднять по короткой инструкции
- новый человек может повторить запуск без нестандартных обходов
- конфигурация читается предсказуемо
- приложение стартует стандартно

# Блок D. Провести ручное auth-тестирование

## Цель блока

Проверить не только код по файлам, а реальное поведение auth-flow как работающего backend-модуля.

## Нужно проверить сценарии

### D.1. Регистрация нового пользователя в новую семью

Ожидается:

- пользователь создаётся
- семья создаётся
- роль назначается как `admin`

### D.2. Регистрация пользователя в уже существующую семью

Ожидается:

- новый пользователь создаётся
- семья не дублируется
- роль назначается как `user`

### D.3. Логин с корректным паролем

Ожидается:

- выдаются access и refresh token

### D.4. Логин с неверным паролем

Ожидается:

- управляемая ошибка
- не `500`

### D.5. Refresh по валидному refresh token

Ожидается:

- выдаётся новый access token

### D.6. Refresh по revoked token

Ожидается:

- управляемая ошибка
- не `500`

### D.7. Logout

Ожидается:

- refresh token отзывается
- endpoint возвращает контролируемый ответ

### D.8. `GET /auth/me` по валидному access token

Ожидается:

- возвращается текущий пользователь

### D.9. Password reset request

Ожидается:

- внешний ответ нейтральный
- flow запускается без раскрытия существования пользователя

### D.10. Password reset confirm

Ожидается:

- пароль меняется
- reset token помечается использованным

### D.11. Повторное использование reset token

Ожидается:

- контролируемая ошибка

### D.12. Вход со старым и новым паролем после reset

Ожидается:

- старый пароль больше не работает
- новый пароль работает

## Критерий готовности блока D

Блок считается закрытым, если:

- все сценарии проходят
- либо возвращают ожидаемую контролируемую ошибку
- нет runtime-падений на штатных use case

# 8. Дополнительные замечания по текущему состоянию проекта

Этот раздел не является отдельным блоком работ, но фиксирует важные наблюдения, которые нужно держать под контролем при выполнении ТЗ 3.

## 8.1. `FamilyResponse`

Если endpoint возвращает ORM-объект `Family`, то `FamilyResponse` должен быть готов принимать данные из ORM-объекта, так же как `UserResponse` и `MeResponse`.

## 8.2. `register_family` и `created_by`

Поле `created_by` в модели семьи требует дополнительной проверки на согласованность:

- модели
- SQL init-схемы
- репозитория создания семьи

Это не основной предмет этапа, но при тестировании регистрации этот момент должен быть перепроверен.

## 8.3. `validate_user_by_email`

Нужно внимательно проверить имена аргументов и общую аккуратность сигнатур, чтобы не оставлять мелкие технические несогласованности.

## 8.4. `backend/Dockerfile`

Файл пока фактически пустой. В рамках ТЗ 3 достаточно хотя бы определить, считается ли Docker-запуск частью текущего acceptance-критерия, или на этом этапе штатным считается только локальный запуск backend. Это решение нужно зафиксировать явно.
