Что такое repository

Сначала короткая теория.

Repository — это слой доступа к данным

Он нужен, чтобы не писать SQL и работу с сессией прямо в endpoint или service.

То есть repository отвечает на вопросы вроде:
 • найди пользователя по email
 • создай пользователя
 • обнови пароль
 • сохрани refresh token

⸻

Почему нельзя писать это сразу в API

Если ты начнёшь делать так:
 • endpoint принял запрос
 • там же открыл сессию
 • там же сделал запрос в БД
 • там же создал пользователя
 • там же сделал commit

то получится каша.

⸻

Правильное разделение

API -> Service -> Repository -> DB

API

Сразу готовый каркас

Скопируй это, а потом я разберу построчно.

from sqlalchemy.orm import Session

from app.models.user import User, UserRole

class UserRepository:
    def __init__(self, db: Session):
        self.db = db

    def get_user_by_email(self, email: str) -> User | None:
        return self.db.query(User).filter(User.email == email).first()

    def create_user(
        self,
        email: str,
        password_hash: str,
        role: UserRole,
        family_id: int,
    ) -> User:
        user = User(
            email=email,
            password_hash=password_hash,
            role=role,
            is_active=True,
            family_id=family_id,
        )

        self.db.add(user)
        self.db.commit()
        self.db.refresh(user)

        return user

1. Импорт Session

from sqlalchemy.orm import Session

Что это такое

Session — это объект, через который SQLAlchemy работает с базой.

Через него ты можешь:
 • делать запросы
 • добавлять записи
 • удалять записи
 • коммитить изменения

⸻

1. Импорт модели User

from app.models.user import User, UserRole

Почему импортируем User

Потому что repository будет работать именно с таблицей users, представленной как ORM-модель User.

Почему импортируем UserRole

Потому что при создании пользователя поле role должно получать значение enum-типа.

Как это выглядит мысленно

Если в таблице есть:

и ты вызываешь:

то получишь объект User.

Если вызываешь:

то получишь None.

⸻

Что принимает метод

⸻

Почему здесь password_hash, а не password

Потому что repository не должен заниматься хешированием.

Это очень важный архитектурный принцип

Repository работает с данными,
а не решает бизнес-логику безопасности.

То есть:
 • service хеширует пароль
 • repository сохраняет уже готовый hash

Создание объекта User

user = User(
    email=email,
    password_hash=password_hash,
    role=role,
    is_active=True,
    family_id=family_id,
)

⸻

Что происходит

Мы создаём Python-объект ORM-модели User.

Пока это ещё не “запись в базе”.
Пока это просто объект в памяти Python.

⸻

Почему is_active=True задаём явно

Потому что при создании пользователя мы хотим, чтобы он был активен сразу.

Да, в модели у тебя уже есть default=True, но явное указание здесь нормально и делает код более читаемым.

⸻

Добавление в сессию

self.db.add(user)

Это значит:

SQLAlchemy, возьми этот объект и подготовь его к вставке в БД

Но после add() запись ещё не обязательно физически сохранена.

⸻

Коммит

self.db.commit()

Это очень важная строка.

Она означает:

зафиксировать изменения в базе данных

После commit() пользователь реально будет записан в PostgreSQL.

⸻

Почему нельзя забывать commit()

Если ты сделал:

self.db.add(user)

но не сделал:

self.db.commit()

то запись может не сохраниться.

⸻

Обновление объекта после коммита

self.db.refresh(user)

Что делает refresh

После коммита SQLAlchemy перечитывает объект из БД.

Это нужно, чтобы объект получил значения, которые выставила сама база.

Например:
 • id
 • created_at
 • updated_at

⸻

Почему это полезно

Если ты создаёшь пользователя и сразу хочешь вернуть его дальше в service, то после refresh(user) у объекта уже будет реальный id.

⸻

Возврат результата

return user

То есть метод возвращает уже сохранённого пользователя.

⸻

Как это будет использоваться позже

Примерно так:

repo = UserRepository(db)

existing_user = repo.get_user_by_email("<user@example.com>")

if existing_user is None:
