
Что мы сейчас строим

Тебе нужен файл, который будет отвечать за 3 вещи:

 1. подключение к PostgreSQL
 2. создание “базового класса” для ORM-моделей
 3. создание сессий для работы с БД

В SQLAlchemy это стандартная схема: create_engine() создаёт “двигатель” подключения, sessionmaker() создаёт фабрику сессий, а declarative_base() даёт базовый класс, от которого наследуются ORM-модели. Это базовый паттерн ORM в официальной документации SQLAlchemy.  ￼

⸻

Сначала теория простыми словами

Что такое engine

engine — это не “соединение с БД прямо сейчас”, а объект, который умеет подключаться к базе и управлять соединениями.
Представь его как диспетчера доступа к PostgreSQL. Когда приложению понадобится база, оно попросит у engine соединение. Это соответствует тому, как SQLAlchemy описывает create_engine() и дальнейшую работу через Session.  ￼

Что такое Session

Session — это уже рабочий инструмент, через который твой код будет:
 • читать данные,
 • добавлять строки,
 • изменять записи,
 • делать commit/rollback.

Если говорить очень грубо:
engine = “доступ к кухне”,
session = “рабочий стол, на котором ты готовишь”.
В документации SQLAlchemy Session описывается как основной ORM-объект для разговора с базой.  ￼

Что такое Base

Base — это общий родитель для всех ORM-моделей.
Когда ты потом напишешь класс User(Base), SQLAlchemy поймёт:
 • это ORM-модель,
 • у неё будет таблица,
 • её поля соответствуют колонкам БД.

Это и есть declarative mapping в SQLAlchemy.  ￼

Почему мы не пишем модели прямо сейчас

Потому что модели должны на что-то опираться.

Сначала нужен фундамент:
 • строка подключения,
 • engine,
 • SessionLocal,
 • Base.

И только потом:
 • User,
 • Family,
 • RefreshToken и так далее.

Если начать с моделей без Base и без нормального подключения, получится каша.

Как выглядит строка подключения к PostgreSQL

Для SQLAlchemy она обычно выглядит так:

postgresql+psycopg://USER:PASSWORD@HOST:PORT/DBNAME

Это зависит от драйвера. Для PostgreSQL SQLAlchemy использует диалект PostgreSQL, а дальше нужен DBAPI-драйвер. В современных проектах часто используют psycopg/psycopg3, и строка подключения задаётся через SQLAlchemy URL.  ￼

Для твоего локального docker-сценария логика будет такой:
 • пользователь: home_secretary
 • пароль: твой пароль из .env
 • хост: localhost или имя сервиса db — зависит от того, откуда подключаешься
 • порт: 5432
 • база: home_secretary

Очень важный момент про host

Если Python-приложение запускается:

снаружи Docker, на твоём компьютере

то host обычно будет:

localhost

внутри Docker Compose, в другом контейнере

то host должен быть:

db

Потому что db — это имя сервиса в compose-сети.

Сейчас, пока ты не запускаешь всё целиком, можно временно взять localhost.
