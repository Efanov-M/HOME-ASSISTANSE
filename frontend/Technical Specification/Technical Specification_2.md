
ТЗ 2 — Frontend CSS Base MVP для проекта Home Secretary

1. Смысл этапа

На этом этапе мы не переделываем HTML-структуру заново и не подключаем backend.

HTML-скелет уже собран на предыдущем этапе. Теперь задача — добавить первый слой оформления, чтобы страницы Home Secretary:
 • выглядели аккуратно;
 • читались легче;
 • имели единый стиль;
 • были готовы к дальнейшему усложнению интерфейса.

ТЗ 2 нужно не для “красивого дизайнерского сайта”, а для того, чтобы поверх уже существующего HTML-каркаса собрать первую системную CSS-основу проекта.

После завершения этого этапа у проекта появится:
 • единый базовый стиль;
 • читаемая структура страниц;
 • оформление форм, кнопок, ссылок, карточек и блоков;
 • основа, на которую потом можно будет добавлять:
 • JavaScript;
 • интеграцию с backend;
 • состояния загрузки;
 • более сложный UI.

⸻

1. Что уже считается сделанным

К этому моменту у проекта уже есть:
 • папка frontend/;
 • HTML-страницы в frontend/pages/;
 • папка frontend/static/;
 • базовая навигация между страницами;
 • страницы:
 • index.html
 • login.html
 • register.html
 • dashboard.html
 • tasks.html
 • task-create.html
 • task-detail.html  ￼

Значит сейчас задача не “делать frontend с нуля”, а подключить и выстроить CSS-слой поверх уже существующего HTML.

⸻

1. Главная цель этапа

После завершения ТЗ 2 frontend должен:
 • иметь подключённые CSS-файлы;
 • выглядеть как единый продукт, а не как набор сырых HTML-страниц;
 • сохранять простой и учебный стиль;
 • быть пригодным для дальнейшей интеграции с JS и backend без переписывания структуры.

⸻

1. Что входит в ТЗ 2

В ТЗ 2 входит:
 • создание базовой CSS-структуры;
 • подключение CSS ко всем страницам;
 • единый стиль для:
 • body;
 • заголовков;
 • текста;
 • секций;
 • форм;
 • кнопок;
 • ссылок;
 • списков задач;
 • карточки задачи;
 • выравнивание отступов;
 • ограничение ширины основного контента;
 • базовое визуальное разделение блоков;
 • подготовка стилей под будущие состояния задач.

⸻

1. Что в ТЗ 2 не входит

На этом этапе не входит:
 • JavaScript;
 • fetch;
 • работа с API;
 • отправка форм;
 • авторизация;
 • токены;
 • динамическая подстановка данных;
 • адаптивность под все устройства на продакшн-уровне;
 • анимации;
 • модальные окна;
 • dropdown-меню;
 • dark mode;
 • сложная дизайн-система;
 • Tailwind, Bootstrap и другие CSS-фреймворки.

То есть сейчас только:

базовый собственный CSS-слой поверх HTML-каркаса

⸻

1. Архитектурное правило этапа

CSS на этом этапе должен быть:
 • простым;
 • читаемым;
 • разбитым по смыслу;
 • пригодным для дальнейшего роста.

Запрещено превращать CSS в хаотичный набор случайных правил.

Нужно мыслить так:
 • общие стили;
 • layout;
 • формы;
 • кнопки;
 • страницы задач;
 • служебные классы.

⸻

1. Структура CSS

Во frontend/static/css/ должен появиться как минимум один основной CSS-файл.

Минимально допустимый путь:
 • frontend/static/css/style.css

Допустимо, если ты решишь использовать несколько файлов, но на этом этапе это не обязательно.

Главное требование:
CSS должен подключаться ко всем frontend-страницам единообразно.

⸻

1. Подключение CSS

Каждая HTML-страница должна подключать CSS через <link> в <head>.

Требование:
 • путь должен быть корректным относительно папки pages/;
 • подключение должно быть одинаковым по логике на всех страницах.

⸻

1. Общие стили страницы

Нужно оформить базовый стиль всего документа.

Должно быть настроено:
 • шрифт для всей страницы;
 • цвет текста по умолчанию;
 • цвет фона страницы;
 • базовый line-height;
 • убраны стандартные большие браузерные отступы;
 • единая логика внутренних отступов.

Цель:

Чтобы страницы перестали выглядеть как “сырой HTML из браузера по умолчанию”.

⸻

1. Контейнер и ширина контента

Нужно ввести единый подход к ширине основного содержимого.

Требование:

Основной контент должен:
 • не растягиваться на всю ширину экрана;
 • иметь читаемую максимальную ширину;
 • иметь внутренние отступы слева и справа.

Цель:

Чтобы текст, формы и блоки читались легче.

⸻

1. Header, Main, Footer

Нужно оформить базовые структурные блоки:
 • header
 • main
 • footer

Требования:
 • header должен быть визуально отделён;
 • main должен быть основным рабочим полем страницы;
 • footer должен выглядеть спокойно и не отвлекать.

Не нужен сложный декоративный дизайн.
Нужна только чистая структура.

⸻

1. Заголовки и текст

Нужно привести к единому виду:
 • h1
 • h2
 • p

Требования:
 • заголовки должны отличаться по размеру;
 • между заголовками и текстом должны быть понятные отступы;
 • текст не должен слипаться в один массив.

⸻

1. Ссылки

Нужно оформить ссылки так, чтобы они выглядели как элементы интерфейса, а не как случайный браузерный дефолт.

Требования:
 • единый цвет ссылок;
 • понятное состояние при наведении;
 • ссылки не должны теряться на странице;
 • навигационные ссылки должны читаться ясно.

⸻

1. Кнопки

Нужно ввести единый базовый стиль кнопок.

Кнопки должны иметь:
 • одинаковую высоту/логику отступов;
 • читаемый текст;
 • различимый фон;
 • аккуратную рамку или оформление;
 • курсор при наведении;
 • визуально понятное поведение hover.

На этом этапе нужно оформить:
 • Login
 • Register
 • Save task
 • Edit
 • Delete
 • Back to tasks
 • Back to dashboard

Требование:

Delete должна визуально отличаться от обычных действий.

⸻

1. Формы

Это один из главных блоков ТЗ 2.

Нужно оформить формы на страницах:
 • login.html
 • register.html
 • task-create.html

Нужно оформить:
 • form
 • label
 • input
 • textarea
 • select
 • button

Требования:
 • поля должны стоять в колонку;
 • между полями должны быть одинаковые отступы;
 • label должен быть визуально связан со своим полем;
 • поля должны иметь единый стиль;
 • поля не должны быть слишком узкими или слишком широкими;
 • textarea должна выглядеть как часть общей системы.

Цель:

Чтобы формы выглядели как элементы приложения, а не как сырой браузерный прототип.

⸻

1. Dashboard

Файл:

frontend/pages/dashboard.html

Что нужно сделать по стилю:
 • оформить блоки-ссылки на разделы;
 • сделать центральную страницу визуально читаемой;
 • выделить переходы в:
 • Tasks
 • Calendar
 • Finance
 • Documents

Требование:

Пока это всё ещё простая навигационная страница, но она должна уже выглядеть как “центр приложения”.

⸻

1. Страница списка задач

Файл:

frontend/pages/tasks.html

Что нужно сделать по стилю:
 • оформить список задач не как голый текст, а как набор блоков;
 • каждая задача должна выглядеть как отдельный элемент;
 • title должен быть заметнее;
 • status должен быть визуально отделён;
 • ссылка на просмотр задачи должна быть понятной.

Дополнительно:

Нужно подготовить визуальную основу под будущие статусы, например:
 • pending
 • in progress
 • done

На этом этапе можно ограничиться разными классами или заготовкой под них, даже если данные пока статические.

⸻

1. Страница создания задачи

Файл:

frontend/pages/task-create.html

Что нужно сделать по стилю:
 • оформить форму как единый блок;
 • сделать поля и кнопку визуально связанными;
 • ссылка “Back to tasks” должна быть понятной и не теряться.

⸻

1. Страница просмотра одной задачи

Файл:

frontend/pages/task-detail.html

Что нужно сделать по стилю:
 • оформить страницу как карточку/блок деталей задачи;
 • title должен быть главным акцентом;
 • остальные поля должны быть выстроены в читаемом порядке;
 • блок действий Edit / Delete / Back to tasks должен быть визуально аккуратным.

Важное требование:

Так как эта страница в будущем будет получать данные из backend, её стиль должен быть построен как стиль для шаблона данных, а не как оформление случайного статического текста.

⸻

1. Визуальное разделение секций

Нужно ввести простую систему отделения блоков друг от друга.

Можно использовать:
 • отступы;
 • границы;
 • лёгкий фон;
 • скругления;
 • тени в очень умеренном виде.

Требование:

Интерфейс должен стать структурированным, но не перегруженным.

⸻

1. Единый стиль карточек и блоков

Нужно определить базовый внешний вид для контентных блоков:
 • секции;
 • формы;
 • задачи;
 • карточки деталей.

Цель:

Чтобы страницы выглядели частями одной системы.

⸻

1. Требования к классам

CSS-классы должны называться понятно.

Допустимая логика:
 • page-title
 • page-description
 • form-group
 • task-card
 • task-status
 • action-buttons

Нельзя использовать хаотичные имена вроде:
 • block1
 • test2
 • aaa
 • my-div-final-real

⸻

1. Требования к CSS-коду

CSS должен быть:
 • читаемым;
 • разделённым по смысловым блокам;
 • с понятными комментариями при необходимости;
 • без дублирования одинаковых правил, если их можно объединить.

⸻

1. Критерии готовности ТЗ 2

ТЗ 2 считается закрытым, если выполнено всё ниже:

 1. во frontend есть подключённый CSS;
 2. все страницы используют единый базовый стиль;
 3. формы оформлены в единой логике;
 4. кнопки оформлены в единой логике;
 5. ссылки оформлены в единой логике;
 6. dashboard.html выглядит как центральная страница приложения;
 7. tasks.html оформлен как список задач, а не как сырой текст;
 8. task-create.html оформлен как нормальная форма;
 9. task-detail.html оформлен как страница просмотра одной задачи;
 10. frontend после этого этапа визуально стал цельным, но при этом остался простым и пригодным для следующих этапов.

⸻

Правильный порядок работы на этом этапе такой:

Шаг 1

 • body

 • текст
 • ссылки

Шаг 3

 • кнопки
 • формы
 • поля ввода
 • textarea
 • select

Шаг 4

 • dashboard.html
 • tasks.html

Шаг 5

Оформить:
 • task-create.html
 • task-detail.html

Шаг 6

Проверить, что все страницы выглядят как части одного интерфейса

⸻

1. Что я считаю самым важным на этом этапе

Главная задача ТЗ 2 — не “навести красоту”, а ввести первый визуальный порядок.

После этого этапа у тебя должен появиться не просто набор HTML-файлов, а уже первый frontend приложения, который:
 • не стыдно открывать;
 • удобно читать;
 • удобно дальше развивать;
 • логично подключать к JS и backend.

⸻

1. Короткий итог

ТЗ 1 дало тебе HTML-скелет.
ТЗ 2 должно дать тебе первый CSS-слой поверх этого скелета.

Именно после этого frontend Home Secretary перейдёт из состояния:

“набор учебных HTML-страниц”

в состояние:

“первая оформленная frontend-основа проекта”.
