ТЗ 1 — HTML Frontend Skeleton для проекта Home Secretary

1. Смысл этапа

На этом этапе не нужно писать полноценный frontend.

Задача этапа:
создать статический HTML-каркас будущего интерфейса проекта Home Secretary, чтобы:

- визуально увидеть структуру будущего приложения;
- начать развивать frontend-навыки без перегрузки JavaScript и backend-интеграцией;
- подготовить основу для дальнейшего подключения CSS, JavaScript и API.

Итог этапа:

- существует отдельная frontend-структура проекта;
- созданы основные HTML-страницы;
- каждая страница имеет понятную структуру и назначение;
- навигация между базовыми страницами продумана.

⸻

1. Что уже считается готовым в проекте

На момент старта frontend-этапа уже существует backend-часть проекта Home Secretary.

Уже реализовано или подготовлено:

- FastAPI backend;
- auth-модуль;
- users / families;
- базовый модуль tasks в проектировании;
- PostgreSQL;
- docker-запуск backend + db;
- README и технические задания проекта;
- архитектура modular monolith.

Это означает:

- frontend пока не должен спорить с backend-архитектурой;
- HTML-каркас должен учитывать уже существующие модули:
  - auth
  - dashboard
  - tasks
  - calendar
  - finance
  - documents

Но на текущем этапе frontend не обязан работать с API.

⸻

1. Главная цель этапа

После завершения ТЗ 1 должны существовать:

1. frontend-папка проекта;
2. понятная иерархия директорий;
3. базовые HTML-страницы;
4. единая структура страниц:
   - header
   - main
   - footer
5. базовая навигация между страницами;
6. текстовый и визуальный каркас для будущего развития интерфейса.

⸻

1. Что входит в ТЗ 1

В этот этап входит:

- создание frontend-структуры;
- создание HTML-страниц;
- описание назначения каждой страницы;
- размещение основных текстовых блоков;
- размещение базовых форм;
- размещение базовых ссылок и кнопок;
- заготовка навигации.

⸻

1. Что не входит в ТЗ 1

На этом этапе не входит:

- CSS-дизайн;
- адаптивность;
- JavaScript;
- Fetch API;
- работа с backend;
- токены;
- localStorage;
- React;
- компонентная система;
- шаблонизация;
- валидация форм через JS;
- анимации;
- тёмная тема;
- иконки и декоративная графика.

То есть этап 1 — это только:
статический HTML-каркас.

⸻

1. Архитектурное правило этапа

Frontend сейчас строится как отдельная учебная часть.

Нужно придерживаться принципа:

- сначала структура;
- потом страницы;
- потом стили;
- потом логика;
- потом интеграция с backend.

Запрещено:

- пытаться сразу делать “красивый готовый интерфейс”;
- смешивать HTML, JS и backend-задачи на этом этапе;
- делать страницы без понятного назначения.

⸻

1. Иерархия папок

Нужно создать frontend-часть проекта с такой базовой структурой:

frontend/

- pages/
- static/
- static/css/
- static/js/
- static/img/

Назначение папок:

1. frontend/

- корень frontend-части.

1. frontend/pages/

- все HTML-страницы проекта.

1. frontend/static/

- общая папка для статических файлов.

1. frontend/static/css/

- будущие CSS-файлы.

1. frontend/static/js/

- будущие JS-файлы.

1. frontend/static/img/

- изображения, логотипы, иконки и прочие медиафайлы.

На этом этапе допускается, что папки css/js/img будут пока почти пустыми.

⸻

1. Базовые страницы этапа

На первом этапе нужно создать следующие страницы:

1. index.html
2. login.html
3. register.html
4. dashboard.html
5. tasks.html
6. task-create.html
7. task-detail.html

Этого достаточно, чтобы собрать первый каркас интерфейса проекта.

⸻

1. Общая структура каждой страницы

Каждая страница должна иметь общую базовую структуру:

- DOCTYPE
- html
- head
- title
- body

Внутри body:

- header
- main
- footer

Цель:
сразу приучиться к нормальной и единообразной структуре документа.

⸻

1. Блок A. Главная страница

Файл:

- frontend/pages/index.html

Назначение:
главная входная страница проекта.

Что должна содержать:

- название проекта: Home Secretary;
- краткое описание проекта;
- короткий текст о том, что это система для организации личных и семейных дел;
- ссылки:
  - Login
  - Register

Страница пока не должна быть перегружена.

Минимально:

- header с названием;
- main с коротким описанием;
- footer с техническим текстом или подписью.

⸻

1. Блок B. Страница входа

Файл:

- frontend/pages/login.html

Назначение:
страница входа пользователя в систему.

Что должна содержать:

- заголовок страницы;
- короткий поясняющий текст;
- форму входа.

Форма должна содержать:

- поле email;
- поле password;
- кнопку Login.

Дополнительно:

- ссылка на Register;
- ссылка Forgot password.

Пока форма статическая, без JS и backend.

⸻

1. Блок C. Страница регистрации

Файл:

- frontend/pages/register.html

Назначение:
страница регистрации нового пользователя.

Что должна содержать:

- заголовок страницы;
- короткий текст;
- форму регистрации.

Форма должна содержать:

- поле email;
- поле password;
- поле family name;
- кнопку Register.

Дополнительно:

- ссылка на Login.

⸻

1. Блок D. Dashboard

Файл:

- frontend/pages/dashboard.html

Назначение:
главная внутренняя страница после входа.

Что должна содержать:

- приветственный заголовок;
- короткий текст о том, что это панель управления Home Secretary;
- ссылки или карточки разделов:

  - Tasks
  - Calendar
  - Finance
  - Documents

На этом этапе часть ссылок может быть заглушкой.

Цель страницы:
показать, как пользователь будет переходить к основным модулям системы.

⸻

1. Блок E. Список задач

Файл:

- frontend/pages/tasks.html

Назначение:
страница со списком задач.

Что должна содержать:

- заголовок Tasks;
- короткий поясняющий текст;
- кнопку Create task;
- простой блок фильтрации;
- список задач.

На этом этапе список задач может быть статическим HTML-списком или набором блоков.

Для каждой задачи достаточно показать:

- title;
- status;
- тип задачи:
  - personal
  - family

Дополнительно:

- ссылка на task-detail.html

⸻

1. Блок F. Создание задачи

Файл:

- frontend/pages/task-create.html

Назначение:
страница создания новой задачи.

Что должна содержать:

- заголовок Create task;
- короткое пояснение;
- форму создания задачи.

Форма должна содержать:

- title;
- description;
- выбор типа задачи:
  - personal
  - family
- due date;
- кнопку Save task.

На этом этапе форма статическая.

⸻

1. Блок G. Просмотр одной задачи

Файл:

- frontend/pages/task-detail.html

Назначение:
страница просмотра одной задачи.

Что должна содержать:

- title задачи;
- description;
- status;
- task type;
- due date;
- created by / assigned to — пока можно текстом-заглушкой;
- кнопки:
  - Edit
  - Delete
  - Back to tasks

На этом этапе значения могут быть подставлены статически.

⸻

1. Навигация между страницами

Нужно сразу предусмотреть базовую навигацию:

- index.html
  - Login
  - Register

- login.html
  - Register
  - Forgot password

- register.html
  - Login

- dashboard.html
  - Tasks
  - Calendar
  - Finance
  - Documents

- tasks.html
  - Create task
  - Task detail
  - Back to dashboard

- task-create.html
  - Back to tasks

- task-detail.html
  - Back to tasks

Цель:
сразу приучиться думать интерфейс как связанную систему страниц, а не как отдельные несвязанные файлы.

⸻

1. Текстовое наполнение страниц

На первом этапе на каждой странице должны быть:

- понятный заголовок;
- короткий поясняющий текст;
- осмысленные подписи у полей;
- названия кнопок на английском языке.

Это нужно, чтобы HTML был не “пустым каркасом”, а уже напоминал реальный интерфейс.

⸻

1. Критерии готовности ТЗ 1

Этап считается закрытым, если:

1. создана frontend-структура папок;
2. созданы базовые HTML-страницы;
3. у каждой страницы есть:
   - title
   - header
   - main
   - footer
4. у страниц есть осмысленный текст;
5. формы login/register/task-create существуют;
6. навигация между страницами продумана;
7. HTML-каркас не смешан с JS и backend-логикой.

⸻

1. Рекомендуемый порядок выполнения

1. создать папки frontend;
1. создать index.html;
1. создать login.html;
1. создать register.html;
1. создать dashboard.html;
1. создать tasks.html;
1. создать task-create.html;
1. создать task-detail.html;
1. проверить структуру и навигацию.

⸻

1. Короткий итог

ТЗ 1 по frontend нужно не для “красивого сайта”, а для того, чтобы собрать учебный и проектный HTML-скелет Home Secretary.

После завершения этого этапа у проекта появится первая frontend-основа, на которую потом можно будет поэтапно добавлять:

- CSS;
- JavaScript;
- интеграцию с backend;
- более сложный UI.
