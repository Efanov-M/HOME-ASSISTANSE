ТЗ 4 — Organizer / Todo Core для проекта Home Secretary

1. Смысл этапа

После завершения этапа стабилизации auth-ядра следующим шагом становится первый прикладной модуль системы — модуль задач.

На этом этапе система должна перестать быть только backend для аутентификации и пользователей и начать работать с реальными пользовательскими сущностями.

Итог этапа:

- пользователь может создавать задачи;
- пользователь может читать свои задачи;
- пользователь может изменять задачи;
- пользователь может удалять задачи;
- задачи могут быть личными или семейными;
- доступ к задачам строится поверх уже существующего auth-механизма;
- все действия проходят через принятую архитектуру:

API -> Service -> Repository -> DB

⸻

1. Что уже считается готовым

На момент старта ТЗ 4 уже существуют:

- FastAPI backend;
- env-конфигурация;
- database.py;
- auth-модуль;
- users / families;
- audit logs;
- docker-запуск backend + PostgreSQL;
- базовая modular monolith структура проекта.

Это означает:

- новый этап не должен ломать существующий auth-flow;
- доступ к задачам должен строиться на уже существующих пользователях и семьях;
- текущая архитектура не пересматривается, а расширяется.

⸻

1. Главная цель этапа

После завершения ТЗ 4 должны работать:

1. создание задачи;
2. получение одной задачи;
3. получение списка задач;
4. обновление задачи;
5. удаление задачи;
6. фильтрация задач;
7. разделение задач на:
   - личные;
   - семейные;
8. защита доступа к задачам через текущего пользователя;
9. базовый audit log на ключевые действия с задачами.

⸻

1. Что входит в ТЗ 4

В этот этап входит:

- ORM-модель задачи;
- repository-слой для задач;
- service-слой для задач;
- Pydantic-схемы для задач;
- FastAPI API для задач;
- базовые проверки доступа;
- базовая фильтрация списка задач;
- audit logging ключевых действий;
- подключение нового router в main.py;
- минимальный ручной smoke-test.

⸻

1. Что не входит в ТЗ 4

Чтобы не расползаться в сторону, в этот этап не входят:

- подзадачи;
- комментарии к задачам;
- чек-листы;
- вложения;
- напоминания;
- recurring tasks;
- приоритеты уровня enterprise;
- сложная система ролей и прав;
- frontend;
- calendar integration;
- realtime updates;
- e2e-автотесты;
- soft delete;
- история изменений задачи по полям.

То есть на этом этапе нужен именно Todo Core MVP.

⸻

1. Архитектурное правило этапа

Модуль задач должен строиться в той же архитектуре, что и auth-модуль:

- API принимает HTTP-запрос;
- Service решает бизнес-сценарий;
- Repository работает с БД;
- Model описывает структуру;
- Auth используется как входной контроль.

Запрещено:

- писать SQLAlchemy query в endpoint;
- решать права доступа в repository;
- смешивать auth-логику и task-логику в одном файле;
- переносить доменные проверки в schemas.

⸻

1. Новая доменная сущность

Нужно ввести сущность Task.

Минимальные поля задачи:

- id
- title
- description
- status
- is_family_task
- created_by
- assigned_to
- family_id
- due_date
- created_at
- updated_at

Смысл полей:

- id — идентификатор задачи;
- title — обязательное краткое название;
- description — необязательное подробное описание;
- status — текущее состояние задачи;
- is_family_task:
  - False — личная задача;
  - True — семейная задача;
- created_by — пользователь, создавший задачу;
- assigned_to — пользователь, на которого задача назначена;
- family_id — семья, к которой относится задача, если задача семейная;
- due_date — дедлайн, если он указан;
- created_at / updated_at — технические временные поля.

⸻

1. Базовые бизнес-правила по задачам

На этом этапе нужно зафиксировать следующие правила:

1. Личная задача:

- создаётся конкретным пользователем;
- принадлежит только ему;
- не видна другим пользователям;
- family_id для неё не обязателен.

1. Семейная задача:

- создаётся пользователем внутри его семьи;
- видна пользователям той же семьи;
- family_id обязателен;
- если у пользователя нет семьи, семейную задачу создать нельзя.

1. Пользователь не должен:

- видеть чужие личные задачи;
- менять чужие личные задачи;
- читать задачи другой семьи;
- менять задачи другой семьи;
- удалять задачу без прав доступа.

1. На этом этапе не вводится отдельная модель “владелец/редактор”.
Достаточно базового правила:

- личная задача доступна только владельцу;
- семейная задача доступна в пределах семьи.

⸻

1. Модель статусов

На этом этапе достаточно простого enum:

- open
- in_progress
- done
- cancelled

Требования:

- статус должен быть enum, а не произвольной строкой;
- статус должен быть единым в ORM и в Pydantic-схемах;
- update-сценарий должен принимать только допустимые статусы.

⸻

1. Блок A. ORM-модель задачи

Нужно создать:

- backend/app/models/task.py

И обновить:

- backend/app/models/__init__.py

Модель должна быть согласована по стилю с текущими ORM-моделями проекта.

В модели должны быть:

- __tablename__;
- поля задачи;
- enum для статуса;
- внешние ключи на users;
- при необходимости внешний ключ на families;
- created_at / updated_at в том же подходе, что и остальные модели.

Нужно решить и зафиксировать:

- допускается ли `assigned_to = NULL`;
- допускается ли `description = NULL`;
- допускается ли `due_date = NULL`.

Критерий готовности блока:

- модель существует;
- импортируется без конфликтов;
- согласована с текущей архитектурой БД.

⸻

1. Блок B. Repository слой задач

Нужно создать:

- backend/app/repositories/task_repository.py

Минимальные методы:

- create_task
- get_task_by_id
- list_tasks
- update_task
- delete_task

Дополнительно допустимо:

- list_personal_tasks
- list_family_tasks

Требования к repository:

- не решает бизнес-логику;
- не проверяет права;
- не определяет, кто может видеть задачу;
- не знает, что такое “текущий пользователь”;
- только читает и пишет данные.

Критерий готовности блока:

- repository умеет выполнять базовые CRUD-операции для задач;
- методы не содержат доменной логики доступа.

⸻

1. Блок C. Service слой задач

Нужно создать:

- backend/app/services/task_service.py

Минимальные сценарии:

- create_task
- get_task
- list_tasks
- update_task
- delete_task

Именно service должен решать:

- задача личная или семейная;
- можно ли текущему пользователю её видеть;
- можно ли её менять;
- можно ли её удалять;
- какие данные допустимы на create/update;
- какой audit писать.

Минимальные бизнес-правила service-слоя:

1. create_task
Должно работать:

- принять текущего пользователя;
- принять данные задачи;
- определить тип задачи;
- если задача семейная — привязать её к семье пользователя;
- если задача личная — ограничить видимость пользователем;
- провалидировать assigned_to, если оно указано;
- записать audit log;
- вернуть созданную задачу.

1. get_task
Должно работать:

- найти задачу по id;
- проверить право доступа текущего пользователя;
- вернуть задачу.

1. list_tasks
Должно работать:

- возвращать список задач, доступных текущему пользователю;
- поддерживать базовую фильтрацию.

1. update_task
Должно работать:

- найти задачу;
- проверить право доступа;
- обновить только разрешённые поля;
- записать audit log;
- вернуть обновлённую задачу.

1. delete_task
Должно работать:

- найти задачу;
- проверить право доступа;
- удалить задачу;
- записать audit log;
- вернуть результат удаления.

Критерий готовности блока:

- все 5 сценариев реализованы;
- доступ проверяется на service-уровне;
- service возвращает предсказуемые результаты и контролируемые ошибки.

⸻

1. Блок D. Схемы

Нужно создать:

- backend/app/schemas/task.py

Минимальные схемы:

- TaskCreateRequest
- TaskUpdateRequest
- TaskResponse
- TaskListResponse

TaskCreateRequest должен включать:

- title
- description
- is_family_task
- assigned_to
- due_date

TaskUpdateRequest должен включать:

- title
- description
- status
- assigned_to
- due_date

Требования:

- update должен быть partial-update friendly;
- response должен быть согласован с ORM;
- response-схема должна поддерживать работу с ORM-объектом так же, как это уже сделано в user-схемах.

Нужно отдельно продумать:

- будет ли `TaskListResponse` содержать:
  - просто список задач;
  - или список + count.
На этом этапе достаточно простого варианта, без пагинации.

Критерий готовности блока:

- схемы описывают вход и выход API;
- бизнес-логики внутри схем нет.

⸻

1. Блок E. API слой

Нужно создать:

- backend/app/api/tasks.py

Endpoint’ы:

1. POST /tasks
2. GET /tasks
3. GET /tasks/{task_id}
4. PATCH /tasks/{task_id}
5. DELETE /tasks/{task_id}

Правила API-слоя:

- endpoint принимает request schema;
- получает db через get_db;
- определяет текущего пользователя через уже существующий auth-механизм;
- вызывает service;
- возвращает response schema.

В API запрещено:

- писать SQLAlchemy query;
- решать права доступа;
- валидировать бизнес-правила задачи;
- напрямую писать audit.

Критерий готовности блока:

- API задач существует как отдельный router;
- router не смешан с auth.py;
- каждый endpoint делегирует логику в service.

⸻

1. Блок F. Доступ и безопасность

На этом этапе нужно обеспечить минимум контроля доступа.

Обязательные правила:

1. Нельзя читать чужие личные задачи.
2. Нельзя изменять чужие личные задачи.
3. Нельзя читать семейные задачи другой семьи.
4. Нельзя изменять семейные задачи другой семьи.
5. Нельзя удалять задачу без прав доступа.

Если правило нарушено:

- service должен выбрасывать контролируемую доменную ошибку;
- API должен возвращать корректный HTTP-ответ;
- недопустимо отдавать 500 на штатный сценарий запрета доступа.

Критерий готовности блока:

- попытка доступа к чужим задачам даёт контролируемый отказ.

⸻

1. Блок G. Фильтрация задач

На этом этапе достаточно базовой фильтрации списка задач.

Минимально допустимые фильтры:

- по status;
- по is_family_task;
- по assigned_to;
- по due_date наличию или диапазону — только если это не усложняет этап слишком сильно.

Важно:

- фильтрация не должна ломать базовый CRUD;
- если фильтрация усложняет этап, можно ограничиться:
  - status;
  - is_family_task.

Критерий готовности блока:

- список задач можно получить не только целиком, но и с базовым отбором.

⸻

1. Блок H. Audit logging

Нужно писать audit хотя бы на:

- создание задачи;
- обновление задачи;
- удаление задачи.

Минимум в details:

- task_id
- title
- is_family_task

Допустимо также добавлять:

- status
- assigned_to
- family_id

Важно:

- audit должен строиться по той же модели, что и в auth-ядре;
- error-path и success-path не должны ломать транзакционную логику.

Критерий готовности блока:

- ключевые операции с задачами фиксируются в audit_logs.

Нужно:

- подключить новый router в backend/main.py;
- убедиться, что auth API и tasks API разделены;
- не нарушить существующую структуру проекта.

После интеграции должны быть доступны:

- /auth/...
- /tasks...

Критерий готовности блока:

- приложение поднимается с новым модулем без конфликта импортов и router-ов.

⸻

1. Минимальный ручной smoke-test

Нужно проверить:

1. пользователь создаёт личную задачу;
2. пользователь видит свою личную задачу;
3. другой пользователь не видит чужую личную задачу;
4. пользователь создаёт семейную задачу;
5. член той же семьи видит семейную задачу;
6. пользователь из другой семьи не видит её;
7. задача переводится из open в in_progress;
8. задача переводится в done;
9. задача удаляется;
10. audit log создаётся на create/update/delete.

1. Что я считаю самым важным в этом этапе

На этом этапе главная цель — не написать “много кода”, а показать, что ты умеешь переносить уже освоенную архитектуру на новый бизнес-модуль.

То есть:

- не повторять auth;
- не смешивать слои;
- не писать todo-логику в endpoint;
